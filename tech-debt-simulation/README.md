
#  Tech Debt Simulation: Engineering Review

##  Context

You have just joined a company as a software engineer.

On your first day, you are assigned an existing codebase built by a previous developer. The system is functional—but not well engineered.

Your task is **not just to read the code**, but to **analyze it like an engineer**, identify problems, and propose improvements.

---

##  Getting Started

### 1. Clone the Repository

```bash
git clone <REPO_LINK>
cd tech-debt-simulation
````

---

### 2. Run the Application

```bash
npm install
node app.js
```
---

## 🔍 Your Task

You are expected to perform a **codebase review**.

### Identify and document:

* ❌ Poor design decisions
* ⚠️ Potential bugs or risks
* 🐢 Performance issues
* 🔐 Security concerns
* 🧱 Missing architectural layers

👉 Minimum: **5–8 meaningful findings**

---

##  Think Like an Engineer

For each issue, answer:

* What is the problem?
* Why is it a problem?
* What is the impact (especially at scale)?
* How would you fix it?

---

## 🛠️ Deliverable

Create your **own GitHub repository** and submit your analysis.

### Your repo should contain:

```md
# Engineering Review

## 📌 Repository Analyzed
[Paste link to this repo]

## 🔍 Issues Found

### 1. Issue Title
- Problem:
- Impact:
- Fix:

### 2. Issue Title
...

## 🏗️ Proposed Improvements
- Architecture changes
- Code structure improvements
- Best practices to introduce

## ⚡ Learnings
- What did you understand from this exercise?
```

---

##  Team Guidelines

* Work in teams of **3–4 members**
* Submit **one repository per team**
* Ensure all members understand the solution

---

##  Important Notes

* Do **not** focus on syntax-level issues
* Focus on **system design and engineering decisions**
* You are not expected to fix everything—**focus on thinking clearly**

---

##  Goal

This exercise is designed to help you:

* Read and understand real-world codebases
* Identify hidden issues in systems
* Think beyond “writing code”
* Develop an engineering mindset

---

##  Hint

> A working system is not necessarily a well-designed system.

---

## 🚀 Extension (Optional)

If you finish early:

* Refactor the codebase
* Introduce proper architecture (routes, services, validation)
* Improve error handling and scalability
